# Chapter 3 - Lab 3: Testing the RAG (Retrieval-Augmented Generation) Concept

The purpose of this lab is to compare the difference between AI responses generated without external context or documents (Context-Free) and AI responses generated by providing a specific document (Grounded), via a Data Engineering API scenario.

## Scenario: Data Streaming API Configuration Error

### Step 1: Asking Questions Without a Document (Baseline - Context Free)

**Prompt:**
> "I am trying to send data to the Kafka queue using our company's 'DataFlowX' API, but I get the error 'Error 4043: Schema validation failed'. How do I solve this error?"

**AI Output (Gemini):**
> Error 4043 is usually caused by the JSON payload you send to the API not matching the expected schema on the server side. To solve it:
> 1. Make sure the data types in your JSON data are correct (e.g., that you didn't send a String instead of an Integer).
> 2. Check that all required fields are included in the request.
> 3. Review your API documentation to confirm the schema requirements.

**Analysis:**
The answer seems logical, but since it does not know the details of the fictional API I made up called "DataFlowX", it listed general software engineering principles (Hallucination). There is no specific solution, because there is no information specific to this company in the model's weights (parametric memory).

---

### Step 2: Asking Questions by Providing a Document (RAG Simulation)

I asked the model the same question by giving it a piece of documentation from the fictional "DataFlowX" API.

**Prompt:**
> Answer my question by reading the system document below.
> 
> [DOCUMENT - DataFlowX API v2.1]
> Error 4043: Schema validation failed. This error is thrown when the system tries to write data to the Kafka queue but the `event_timestamp` field is not sent in ISO-8601 format (`YYYY-MM-DDTHH:MM:SSZ`). Additionally, with the v2.1 update, the `user_id` field has become mandatory; if it is missing, this error is returned.
> 
> Question: "I am trying to send data to the Kafka queue using the DataFlowX API, but I get the error 'Error 4043: Schema validation failed'. How do I solve this error?"

**AI Output (Gemini - Grounded):**
> To resolve 'Error 4043', you must check the two critical points stated in the documentation:
> 1. Make sure the `event_timestamp` field in the data you send is strictly in ISO-8601 format (e.g., `2026-07-08T15:30:00Z`).
> 2. Verify that you have added the `user_id` field, which became mandatory with API v2.1, to your JSON payload (request).

---

### Step 3: Comparison and Conclusion

I summarize the differences based on "Quality Criteria":

- **Accuracy:** The answer in Step 1 was logical but factually wrong. The answer in Step 2 was 100% correct because it was based on company documentation.
- **Specificity:** While Step 1 said "look at JSON format", Step 2 pinpointed exactly the `"event_timestamp"` and `"user_id"` variables.
- **Grounding:** While the first answer relied entirely on the model's memory, the second answer was produced entirely bound (grounded) to the provided text.
- **Hallucination Risk:** High in the first case. The model could have led the developer to search for errors in the wrong places for hours by generating general advice in a very professional tone for an API it did not know.

**Reliability Reflection (Real Business Context):**
> In a real corporate scenario, especially in DevOps, Finance, or Customer Support processes, **I would solely use the approach in Step 2 (RAG-supported).** Because in a world where corporate information is updated every day, relying on the model's own memory can have disastrous consequences; RAG architecture takes the model out of being a "memorizer" and turns it into an "analyst" relying on external evidence.
